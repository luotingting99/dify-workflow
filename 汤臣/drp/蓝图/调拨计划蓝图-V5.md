# 调拨计划蓝图

## 一、现状运营流程（时序图）

以下时序图展示了运营、本地系统、OCS系统之间的交互关系，基于现网的实际业务逻辑。

```mermaid
sequenceDiagram
autonumber
participant S as 计划部
participant P as 调拨计划专员
participant SYS as 本地
participant D as OCS系统


note over S,D: 【数据收集和整合】
rect rgb(245, 253, 252)
loop 每月

S->>P: 计划部提供月度销售预测数据及各物理仓预测数据
note right of S: 提供销售预测数据
end 

loop 每日
note right of P: 调拨计划专员从OCS系统导出数据到本地excel
P->>D: 导出即时库存(在库、在途）到本地系统
D-->>P: 返回今日即时库存

P->>D: 导出历史（导预测月上一自然月）销售明细本地系统
D-->>P: 返回历史销售明细


P->>P: 整合数据

end
end


rect rgb(241, 249, 242)
note over S,D: 【需求预测与库存策略计算】

Par 计算RDC仓的需求量（北京仓、昆山仓、成都仓、武汉仓），需求量=物理仓_SKU_需求量_月份
P->>SYS: Excel:计算RDC仓需求量
note over SYS: 需求预测：按计划部销售预测提供的值作为目标库存值
SYS-->SYS: Excel计算目标库存
SYS->>P: 返回目标库存
end

Par 计算RDC仓的SKU是否需要补货
P->>SYS: Excel:计算补货点
note over SYS: 补货ROP策略：按 各仓缺货总量排序，从大到小依次补货，每个仓的SKU按照缺货量从大到小依次补货
SYS-->SYS: Excel计算补货点
SYS->>P: 返回补货列表(物理仓_SKU_补货点_月份)

end

Par 计算库存止拨策略
P->>SYS: Excel: 过滤止拨SKU
note over SYS: 	如果珠海仓库存少于 10000 件，那就一件都不能往外调。如果库存多于 10000 件，那就得先留够自己要用的那一份（全仓库存*销售预测比例），再把剩下的分给其他仓。
SYS-->SYS: Excel计算止拨SKU
SYS->>P: 过滤SKU
end


Par 计算RDC仓的计划补货量
P->>SYS: Excel:计算计划补货量
note over SYS: 补货量策略：需求量 - 即时库存（含在途）
SYS-->SYS: Excel计算计划补货量
SYS->>P: 返回补货列表(物理仓_SKU_计划补货量_月份)
end
end 

rect rgb(252, 242, 248)
note over S,D: 【物流策略和调拨单生成】


Par 选择发货批次和数量
P->>SYS: Excel:选择批次和数量
note over SYS: 批次选择策略：按月份的需求值，选择数量最接近的需求量的批次（月份有效批次），需要整板或整箱调拨
SYS-->SYS: 手工选择批次
SYS->>P: 返回目标SKU的批次和数量
end

Par 计算装车策略
P->>SYS: Excel:相同线路合并，计算装车体积
note over SYS: 装载策略：调拨批次的总体积 ≥100，并 < 160 方 时，才允许发车
note over SYS: 超体积策略：调拨批次的总体积 ＞ 160方 时，执行裁剪，确保不超过车辆体积；具体阈值依据实际情况调整
SYS-->SYS: Excel按线路计算装车体积，手工调整发货批次
SYS->>P: 线路和装车提交

end 



P->>P: 物流约车，确定具体发货时间

P->>SYS: Excel: 构建RDC的调拨单据
SYS->>SYS: Excel:生成调拨单据
SYS-->P: 调拨单明细列表
end 


P->>D: 导入OCS系统

D-->P: 审核通过

```


## 二、 现状存在的问题

## 📌 问题与优化空间

| 问题分类 | 现状问题 | 优化空间 |
|----------|----------|----------|
| **数据收集与整合** | 1. 销售预测与即时库存需人工汇总，操作繁琐、易出错 | 1. 每月将销售预测数据导入 Agent 后，Agent 每天自动同步库存，Agent自动汇总统计数据 |
| **计算 RDC 需求量** | 1. 我们对 2025-02-06 的对照统计显示：预测与实际销售偏差强度约 78%，仅依赖销售预测不足以抵消月内波动。<br/>如果只把“销售预测”作为目标库存，虽然能够覆盖整月的总需求，但无法反映月内的波动，导致 RDC 容易出现缺货。<br/>**例**：成都仓，月销售预测=10,000，但当月实际销售=15,000，超出的 5,000 并未被预测覆盖，只能依靠“安全库存”来做缓冲。 | 1. 因为单纯依赖“销售预测”无法消除月内波动，容易导致 RDC 缺货，所以需要增加安全库存，用来抵消预测的波动性，从而提高整体预测的稳定性和准确性。<br/><br/>2. （二期实现）给商品按销量波动性打标签：低波动类商品可设高服务水平并配较高的安全库存保障供应，高波动类商品只设低服务水平并配较低的安全库存以避免积压。<br/>**例**：某 SKU 每天都卖 200 件，很稳定 → 属于低波动类，适合配高安全库存；某 SKU 有时卖 200 件，有时卖 0 件，忽高忽低 → 属于高波动类，只适合配低安全库存。  |
| **补货策略** | 1. 补货周期：目前按月补货，导致在库水位偏高。当珠海仓库存不足以覆盖全部 RDC 需求时，各 RDC 会按“月目标差额”争抢，容易造成一边过量、一边缺货。<br/>**例**：成都仓在库 = 500，月销售预测 = 4000；若珠海仓某批次有 1500 库存并直接全部补给成都仓，则珠海仓自身库存将低于月销售预测，导致其他 RDC 无法获得补货。<br/><br/>2. 补货时机：人工判断规则“缺货最多优先补”，容易过早或过晚。<br/>**例**：成都仓在库= 500，月销售预测= 4000，按规则需补 3500。但结合日均预测销量 100（假设） 来看，现有库存可支撑 5 天（500 ÷ 100 = 5），而运输提前期仅 3 天，因此仓库还有 2 天缓冲期。理论上可暂缓补货，把资源优先分配给更紧缺的 RDC。<br/><br/>3. 补货数量：现状主要参考批次数量和销售预测，通常一次只补一个批次，补多少取决于批次库存，若只看单批次库存来补货，而不是按需求缺口来算，可能出现补货量偏大或偏小，导致RDC库存被动拉高或仍然缺货。<br/>**例**：成都仓在库 =500，月销售预测= 4000，如果直接用珠海仓单批次 1500 来补货，看似补足，但并不知道能支撑几天。如果成都仓日均销量预测只有 100，这批货可以撑 15 天，就会造成库存积压；如果日均预测销量高达 500，这批货只能撑 3 天，很快又会缺货。 | 1. 补货周期从原来的月改为周，可以降低目标库存。<br/>**例**：成都仓在库=500，月销售预测=4000；若转为周度补货，周均销售预测=1000，则只需补 500，其余 3000 可用于其他 RDC。<br/><br/>2. 增加补货点统计值，当即时库存（含在途） < 补货点时触发补货。<br/>**例**：成都仓在库=500，周销售预测=1000，日均销售预测=100，运输期=3 天；只要库存 ≥300（3×100），就可暂不补货。<br/><br/>3. 增加补货量统计值，防止补多/补少。<br/>**例**：成都仓在库= 500，周销售预测 =1000，只需补足缺口 500，如单批次不够则混合批次，既能满足需求又避免浪费。 |
| **批次策略** | 1. 补货策略与履约策略冲突。现状只补近 3 个月批次，旧批留在珠海仓。导致订单从珠海仓发货，影响本地订单履约率。<br/><br/>2. 调拨近 3 个月的批次，如果 RDC 没有销量，那么新批次也会逐渐变为旧批次。我们对 2025-02～06 的本地履约率统计显示，整体拆单率约为 55%，其中由旧批次驱动的拆单比例达到 32.9%，并且在 2～6 月逐月升高，说明旧批次的累积正在推动拆单率不断上升。 | 1. 旧批次库存按各 RDC 的销售占比分配，确保旧批次在高销量仓优先消化，避免积压或缺货。<br/>**例子**：某 SKU 有旧批次库存 1000 件，北京/昆山/成都 3 个 RDC 的预测销量占比分别为 30% / 50% / 20%。按规则应分配：北京 300 件、昆山 500 件、成都 200 件。 |
| **止拨策略** | 1. 止拨策略采用月预测，使珠海仓保留了高水位库存，导致 RDC 缺货，同时没有加入安全库存来抵消预测波动。<br/>**例**：成都仓在库=500，月销售预测=4000；珠海仓在库=5000，月销售预测=5000；按照规则，不能进行调拨，此时会导致 RDC 缺货。 | 1. 优化珠海仓的止拨数量=周需求预测+安全库存，提高资源利用率。<br/>**例**：成都仓在库=500，周销售预测=1000；珠海仓在库（含待检）=5000，周销售预测=1000，安全库存=500；那么珠海仓的止拨阈值=1500（1000+500），因此有 3500 的富余库存可用于给其他 RDC 补货。 |

---

***说明：**    
- 目标库存：=周转库存+安全库存  
- 周转库存：=周需求预测*周期（比如1周+运输期3天≈1.4周）
- 安全库存：SS = z × σ_pp
  -	**例子**：某 SKU 每周平均卖 100 件，销量有时多 20 有时少 20，历史误差的平均波动幅度约 16 件（σ_PP≈16）。若希望 95% 的订单能被满足（z=1.65），就需要额外准备约 27 件安全库存（=1.65×16）。
  - σ_PP（需求波动）：反映历史预测和实际销量的误差有多大波动，波动越大说明越不稳定，需要更多安全库存。
  - z（服务水平系数）：表示希望多少比例的订单能被满足，比如设定 95% 的服务水平，对应 z=1.65，就是要把库存放大到能覆盖绝大多数需求波动。
  

## 三、改造后的场景及运营流程


### 生成调拨计划
```mermaid
sequenceDiagram
autonumber
participant S as 计划部
participant P as 调拨计划专员
participant A as 调拨计划Agent
participant D as OCS系统

rect rgb(245, 253, 252)
note over S,D: 【数据接入与自动汇总】
loop 每月
P->>A: 输入生成本月需求预测
A-->>P: 返回月度需求预测结果
P->>A: 修改月度需求预测导入Agent
A-->>P: 月度需求预测数据总结报告
note over A,P: 校验报告包含SKU覆盖度、物理仓分布、异常数据清单等
end
loop 每日
A->>D: 触发/查看数据刷新
A->>A: 自动同步销售/价格/库存快照并聚合到物理仓
end
end

rect rgb(241, 249, 242)
note over S,D: 【SKU+仓周度目标库存生成】
P->>A: 人工触发：帮我生成调拨计划
Par 统计周度需求预测（全仓）
note over A: 基于预测模型统计生成目标周需求预测，维度：物理仓_SKU_预测量_周
A->>A: 计算周需求预测

end
Par 统计周转库存（全仓）
note over A: 周转库存 = 周需求预测 × 周期；其中周期 = 1周 + 3天运输周期（约1.43周）；维度：物理仓_SKU_周转库存_周
A->>A: 计算周转库存

end

Par 统计安全库存（全仓）
note over A: 基于历史周度"预测误差"(实际−预测)的标准差，结合覆盖期(评审周期+提前期)与目标服务水平(CSL)，计算安全库存；维度：物理仓_SKU_安全库存_周
A->>A: 计算安全库存

end

Par 统计目标库存（全仓）
note over A: 基于周需求预测、提前期和安全库存，统计目标库存 = 周转库存 + 安全库存；其中周转库存 = 周需求预测 × (评审周期 + 提前期)，维度：物理仓_SKU_目标库存_周。
A->>A: 计算需求预测值
end

Par 统计补货点ROP（全仓）
note over A: 基于周需求预测、安全库存和提前期，统计补货点ROP=提前期库存+安全库存，维度：物理仓_SKU_ROP
A->>A: 计算ROP
end

A-->>P: 生成目标库存表（包含字段：SKU、物理仓、周期起止时间、目标库存、周转库存、周需求预测、提前期、提前期库存、安全库存、服务水平等）
P->>A: 人工审核/微调
end

rect rgb(252, 242, 248)
note over S,D: 【SKU+物理仓的补货计划】
P->>A: 人工触发生成补货计划
Par 触发判定（全仓）
note over A: 若即时库存（在库 + 在途） < 补货点(ROP)，则触发补货；未触发则跳过
A->>A: 判定是否触发补货
end
Par 计算理论补货量（全仓）
note over A: 基于触发结果，计算理论补货量 = 目标库存 − 即时库存。其中：目标库存 = 周转库存 + 安全库存。
A->>A: 计算理论补货量
end

Par 止拨校验（全仓）
note over A: 对调出仓执行止拨校验：在满足止拨阈值后，剩余的库存才允许调拨。（珠海仓）止拨阈值=周转库存（1周） + 安全库存
A->>A: 统计调出仓可调拨量
end

Par 修正补货量（全仓）
note over A: 实际补货量 = min(理论补货量，可调拨量)，并进行修正：包括包装/起订量取整，最终形成调整后的实际补货量
A->>A: 输出修正后的实际补货量
end

A-->>P: 生成补货计划（字段：SKU、物理仓、实际补货量、理论补货量、目标库存、周转库存、安全库存、即时库存、ROP、止拨数量等）及报告（包含物理仓维度调拨体积，调拨总SKU数量，调拨总数量等）
P->>A: 人工审核/微调
end

rect rgb(238, 245, 255)
note over S,D: 【生成调拨计划（SKU_逻辑仓_批次）】

Par 统计珠海仓需调出总量（珠海仓）
note over A: 汇总各补货RDC的缺口需求；珠海仓需调出总量 = 各RDC目标缺口量之和
A->>A: 输出【珠海仓需调出总量（SKU×逻辑仓×数量）】
end

Par 生成批次池
note over A: 在珠海仓调出总量范围内，汇总所有可用批次（SKU、批次、数量、逻辑仓）；当前销售组织仓 → 可直接使用；股份仓 → 判断是否预留给本组织；若无明确标记 → 暂停使用，需人工确认
A->>A: 生成批次池【批次池（SKU×批次×数量×逻辑仓）】
end

Par 分配批次到RDC
note over A: 按各RDC补货需求比例分配批次池中的库存；遵循先进先出（先入库批次优先）；避免只用新批次导致旧批积压
A->>A: 计算【各RDC的分配批次数量】
end

A-->>P: 输出【调拨计划草稿（包含结果+异常提示）】
P->>A: 人工确认（批次可用性、调整数量）
A->>A: 基于确认信息重新计算分配，输出【修正版调拨计划】
A-->>P: 输出【修正版调拨计划（最终版，可执行）】
end
```

### KPI 报告

```mermaid
sequenceDiagram
autonumber
participant P as 调拨计划专员
participant A as 调拨计划Agent
participant D as OCS系统

rect rgb(245, 253, 252)
note over P,D: 【数据接入与预处理】
loop 每日0点
A->>D: 拉取前一天销售明细(订单/仓/批次/发货数量)
D-->>A: 返回前一天销售明细
A->>A: 标准化销售明细数据
A->>A: 匹配目标物理仓映射
end

loop 每月30号
A->>D: 拉取当月物流运费明细（订单/仓/物流运费/物流运费体积）
D-->>A: 返回当月物流运费明细
A->>A: 标准化物流运费数据
end

end

rect rgb(241, 249, 242)
note over P,D: 【定期自动生成报告】
loop 每周一
A->>A: 统计本地订单满足率、拆单率、旧批次拆单率
note over A: 本地订单满足率=本地履约订单数/总订单数；拆单率=跨仓拆单履约订单数/总订单数；旧批次拆单率=旧批次跨仓拆单订单数/旧批次订单数
A->>A: 聚合周度KPI：本地订单满足率/旧批次拆单占比
end

loop 每月初
A->>A: 统计上一自然月的物流运费成本
note over A: 物流运费成本包含：总运费、订单运费、调拨运费、单位物流运费成本以及省级单位物流运费成本
A->>A: 聚合月度物流运费成本报告
end

A->>A: 系统自动生成固定版本报告
end

rect rgb(252, 242, 248)
note over P,D: 【人工触发报告】
P->>A: 人工触发生成报告（指定仓/周/月）
A->>A: 重新计算指定维度的KPI指标
A-->>P: 生成报告或下载
end

rect rgb(238, 245, 255)
note over P,D: 【报告查看与导出】
P->>A: 查看已生成的报告
A->>A: 直接取系统存好的结果（不重新计算）
A-->>P: 下载报告
end
```

## 附录 指标统计
### RDC仓 月度指标（A.A品类，2025-02至2025-06）

- WMAPE（加权平均绝对百分比误差）：衡量预测误差在总需求中的占比，数值越低越好。
- **例子**：这 3 周一共卖了 600 件，预测一共错了 90 件（绝对值）。错误占比 = 90 ÷ 600 = 15%。所以这个预测模型的 加权平均误差率 = 15%。
- MAPE（平均绝对百分比误差）：衡量预测误差的平均水平，不考虑需求规模，容易受小样本影响。
- **例子**：这 3 周卖了 600 件，预测分别错了 30、30、30 件。每周的错误率分别是 30 ÷ 100 = 30%、30 ÷ 200 = 15%、30 ÷ 300 = 10%。所以这个预测模型的 平均误差率 = (30% + 15% + 10%) ÷ 3 = 18.33%。
- 期间偏差率：衡量预测合计与实际合计的整体偏差方向，正值说明预测高估，负值说明预测低估。
- **例子**：这 3 周一共卖了 600 件，预测一共卖了 690 件。期间偏差率 = (690 - 600) ÷ 600 = 15%。说明整体预测高估了 15%。

- 珠海仓普遍高估，容易形成库存积压；北京仓和昆山普遍低估，存在缺货风险；成都仓波动较大，增加了计划和执行的难度。
- 行业经验表明，合理的预测误差应控制在 10%~20% 之间。要提升准确性，需要进一步优化预测口径和方法。

### 安全库存
**例子**：某 SKU 在 3 周内预测误差分别是 0、+20、-20；
平均误差 = (0+20+(-20)) ÷ 3 = 0；
方差 = [(0-0)²+(20-0)²+(-20-0)²] ÷ 3 ≈ 266.7；
标准差 σ_PP = √266.7 ≈ 16。
若希望 95% 的订单能被满足（z=1.65），就需要额外准备约 27 件安全库存（=1.65×16）。
